## The Mock Listings Dataset

`data/listings.json` contains 40 mock secondhand listings across categories (tops, bottoms, outerwear, shoes, accessories) and styles (vintage, y2k, grunge, cottagecore, streetwear, and more).

Each listing has: `id`, `title`, `description`, `category`, `style_tags`, `size`, `condition`, `price`, `colors`, `brand`, and `platform`.

Load it with:
```python
from utils.data_loader import load_listings
listings = load_listings()
```

## The Wardrobe Schema

`data/wardrobe_schema.json` defines the format your agent uses to represent a user's existing wardrobe. It includes:

- `schema`: field definitions for a wardrobe item
- `example_wardrobe`: a sample wardrobe with 10 items you can use for testing
- `empty_wardrobe`: a starting template for a new user

Load an example wardrobe with:
```python
from utils.data_loader import get_example_wardrobe
wardrobe = get_example_wardrobe()
```

## Tool Inventory

**1. search_listings**

Purpose: Searches the mock JSON dataset for thrift items matching specific constraints.

Inputs: - description (str): Keywords describing the item (e.g., "vintage graphic tee").

size (str | None): Size string to filter by (e.g., "M").

max_price (float | None): Maximum price boundary.

Outputs: A list[dict] containing matching listing dictionaries, sorted by keyword relevance score.

**2. suggest_outfit**

Purpose: Acts as a virtual stylist by taking a newly found thrift item and suggesting 1-2 complete outfits.

Inputs:

new_item (dict): The listing dictionary of the item found in step 1.

wardrobe (dict): The user's existing wardrobe dictionary.

Outputs: A str containing a paragraph of specific styling advice and outfit combinations.

**3. create_fit_card**

Purpose: Generates a short, catchy, shareable caption for social media (Instagram/TikTok).

Inputs:

outfit (str): The outfit suggestion string generated by suggest_outfit().

new_item (dict): The listing dictionary for the thrifted item.

Outputs: A str containing a 2-3 sentence casual Instagram caption.

## Error Handling and Fail Points


| Tool | Failure mode | Agent response |
|------|-------------|----------------|
| `search_listings` | No results match the query | The agent catches the empty list, sets session["error"], and halts the loop. During testing, submitting "designer ballgown size XXS under $5" successfully triggered the graceful halt. The Gradio UI cleanly displayed the intended error
 |
| `suggest_outfit` | Wardrobe is empty | Detects the empty list and dynamically alters the LLM prompt to offer general styling advice instead of hallucinating items the user doesn't own |
| `create_fit_card` | utfit input is missing/incomplete | Intercepts the empty string and returns a safe fallback error message ("Error: Outfit suggestion is missing. Cannot generate a fit card.") instead of raising a Python exception |

**One way planning.md helped during implementation:**

Defining the exact inputs and return values for the tools in the planning document made writing the planning loop in agent.py incredibly straightforward. Because I had explicitly planned that search_listings would return an empty list [] on failure (rather than raising an exception), I knew exactly what conditional branch (if not session["search_results"]:) to write to control the agent's flow and gracefully handle errors.

**One divergence from your spec, and why:**

In the initial planning.md architecture, I considered using the Groq LLM to parse the user's natural language query into the description, size, and max_price variables. During implementation, I diverged and used standard Python Regular Expressions (re) in the _parse_query() function instead. I did this because it drastically reduced the latency of the agent by avoiding an entire preliminary network call to Groq, making the UI feel much snappier for the user.

## AI Usage

**Instance 1: Implementing the Search Tool**

What I gave the AI: The Tool 1 spec from my planning.md (inputs, return value, failure mode) and the structure of listings.json.

What it produced: A Python function that filtered by price and size, and counted keyword overlaps to score the results.

What I changed/overrode: The AI's generated code used a strict equality check for size (if item["size"] == size:). I overrode this to use a case-insensitive substring match (if size is not None and size.lower() not in item.get("size", "").lower():) so that a query for size "M" would successfully match a listing legitimately tagged as "S/M" or "M/L".

**Instance 2: Implementing the Planning Loop**

What I gave the AI: My planning.md Architecture flowchart and the skeleton of run_agent() from agent.py.

What it produced: The sequential execution steps mapping the state dictionary to the tool calls.

What I changed/overrode: The AI forgot to add an early return statement after catching the empty search_results list, which would have caused suggest_outfit to crash when it tried to access index [0]. I manually overrode the code to insert return session immediately after the error string assignment to properly halt the flow.
